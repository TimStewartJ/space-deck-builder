Key Design Choices

  • PRO Card embeddings are shared across zones — same card in hand vs trade row uses the same identity vector plus a zone offset.
  • PRO Sum pooling is permutation-invariant (card order doesn't matter in a zone).
  • CON After sum pooling, individual card identities are lost — the actor head can't attend to specific cards when choosing actions.
  • CON Actor outputs a flat logit vector over all possible actions, most of which are masked out. No structural link between “buy card X” and “card X is in trade row.”

Proposed: Attention-Based Actor Head

Replace the flat MLP actor head with query-key dot products. The core idea: for card-indexed actions (play, buy, attack_base), the network attends to specific card embeddings rather than mapping through a giant linear layer.

How Attention Helps in Star Realms

  • Faction synergy: The query can learn to attend to cards matching the current faction composition (e.g., prefer Blob cards when you already have Blob cards in play).
  • Cost awareness: The dot-product score naturally captures “this card's features match what I want to buy given my current resources.”
  • Generalization: A card the model has never seen in a specific slot can still get correct logits if its embedding is close to similar cards.
  • Scalability: Colony Wars or other expansions just need new embeddings — architecture stays fixed.

Incremental Implementation Plan

STEP 1 Attention actor head only — swap the actor's final Linear(128→A) with query-key attention for card-indexed action types. Keep scalar actions (end_turn, skip, attack_player) as a small MLP head. Critic head stays unchanged.

STEP 2 Multi-head attention — use H heads (e.g. 4) so different heads can capture different card relationships (cost vs. faction vs. zone).

STEP 3 Cross-attention in trunk — before the actor head, let zones attend to each other (hand cards attend to trade row for synergy detection).

Common Pattern

All modern game AIs with variable-size entity sets (units, cards, pieces) use some form of entity attention — the policy head attends to individual entities rather than flattening everything into a fixed vector. Our current model is the flat approach; the attention proposal aligns with the state of the art.

Attention in LLMs

LLM Self-Attention

  • Entities: Tokens (words/subwords)
  • Q, K, V: All come from the same sequence
  • Purpose: Every token attends to every other token to build contextual representations
  • Scale: d_model = 4096+, 32+ heads, 80+ layers
  • Output: Next token prediction via final projection

Our Proposed Card Attention

  • Entities: Cards (with zone context)
  • Q: From trunk hidden state (“what I want”)
  • K: From card embeddings (“what's available”)
  • Purpose: Score each card for a specific action type
  • Scale: E = 32, 1-4 heads, 1 layer
  • Output: Per-card action logits (no V needed — just scores)

The Connection

Our card attention is cross-attention — the query comes from a different source (trunk) than the keys (card embeddings). LLMs primarily use self-attention. But the math is identical: score = q · k / √d. The key insight is that dot-product attention is a universal mechanism for “which of these things should I focus on?” — whether “these things” are words in a sentence or cards in a trade row.
